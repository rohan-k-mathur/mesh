## 0. The decision, stated crisply

Session 11 accepted the **diagnosis** (Isonomia's formal core is alethic; its highest-stakes outputs are practical optimality judgments; the inherited Walton `practical_reasoning` scheme is thin) as a standing observation. It did **not** license adopting the spec.

This session takes the engineering reading: **almost all of the spec's *useful* moves survive the framework being dropped**, because they are expressible as incremental upgrades to the Walton machinery already in the repo. The Lumer-specific apparatus is exactly the baggage; the value underneath it is plain scheme/CQ work.

**What we decline (the baggage):** `DesirabilityProfile` / `PracticalNetting` Prisma models; the `practical/pascal` subfamily + `PASCAL_INVALID` fatal-invalidity semantics; partition-invariance as a strengthened validity condition; `MAXIMIN`/`HURWICZ` folds; the separate desirability-monoid register; the static→per-instance CQ-generation migration (deferred — none of the items below require it); and the epistemological-principle fourth-layer *theory* (the conjecture [C016](../03_CONJECTURES/C016-epistemological-principle-fourth-layer.md) stays filed, but no build rides on it).

## 2. The brainstorm — low-baggage enhancements (ranked by value / cost)

Each item drops the framework and keeps the Walton surface. None requires a new Prisma model.

**1. Reconcile the two competing PR seed scripts (pure data; highest value, lowest cost).**
*(Re-scoped by the ground-truth pass.)* The task is not "add CQs to a thin seed" — it is **reconciling two seed scripts that disagree** on the same key: [`schemes.seed.ts`](../../scripts/schemes.seed.ts) (A: 3 CQs, `materialRelation: "practical"`, no slotHints) vs [`seed.practical-reasoning.ts`](../../scripts/seed.practical-reasoning.ts) (B: 6 CQs, `materialRelation: "cause"`, with slotHints). Seed B is the version that ran (audit-confirmed 6-CQ live row) and is the richer, Walton-complete one, so the fix is to **align the stale A block to B (or retire A's `practical_reasoning` entry)** and pick one source of truth — leaving exactly the 6 `PR.*` CQs. This answers the entire "thin scheme" complaint with **zero new abstractions** — no profile, no netting. *(Open sub-decision: `materialRelation` should be `"practical"` (A) on Macagno grounds, not `"cause"` (B) — reconcile to `"practical"` while keeping B's CQs/slotHints.)*

**2. Typed CQ→scheme cross-references (`scheme_reference`).**
Tag the side-effects CQ as pointing to `negative_consequences`; the alternatives CQ as opening rival `practical_reasoning` instances; etc. This is the *de-Lumered* version of "the completeness assumption is itself an argument": a CQ that, when raised, **opens another scheme instance** rather than terminating. **Implemented (2026-06-12) as catalogue-level metadata** ([`lib/schemes/cq-cross-references.ts`](../../lib/schemes/cq-cross-references.ts)) — a static typed edge list, no per-row DB column and no migration, since the edge "PR side-effects → negative_consequences" is a fact about the *catalogue* (identical across every instance and room), not per-instance data. **This is also the concrete handle on [Q-017](../01_OPEN_QUESTIONS_REGISTRY.md#q-017) (CQ-of-CQ termination) — with a corrected finding:** the recursion graph (restricted to `opens` edges) is **finite but NOT a DAG**. `positive_consequences` and `negative_consequences` form a **2-cycle** (NC's "are there offsetting benefits?" → PC, PC's "overlooked harms?" → NC). So termination cannot rely on acyclicity (as the first-pass brainstorm assumed) — it relies on the graph being finite **plus a visited-set over (scheme, claim) pairs** (re-opening a scheme over the same claim is a no-op). The module exposes `analyzeRecursionTermination()` which returns this verdict programmatically.

**3. Expose the goal/value premise as a first-class, contestable slot.**
The whole "expose where evaluation bottoms out" insight (session 11 §4) reduces, without the framework, to: give `practical_reasoning` an explicit `goal`/`value` premise slot (via existing `slotHints`) with its own CQ, and route value-laden arguments to `value_based_pr`. **The "profile" is just the value premise, challengeable as an ordinary premise** — no separate `DesirabilityProfile` table, no `justificationArgumentId` hook, no versioned profile object. This keeps the honest "where does this bottom out" affordance and discards the apparatus. **Ground-truth pass:** the slot already exists (seed B `goal`/`means`/`context` slotHints) and is already contestable (`PR.GOAL_ACCEPTED`, premise-scope `UNDERMINES`); the composer already validates `goal`/`means` roles ([`lib/argumentation/composerEnthymemeCheck.ts`](../../lib/argumentation/composerEnthymemeCheck.ts)). The one genuine gap was **the goal-vs-value *type* that drives routing** — `schemeInference` scored `practical` generically with no instrumental-vs-value distinction. **Implemented (2026-06-12)** as a standalone classifier (item-3 log below).

**4. "Open-by-default, raise-by-instantiation" CQ convention.**
Get the dialectical behaviour the spec wanted from `OPEN_UNREFUTED` **without a new CQ-state enum**: seed the side-effects / alternatives CQs as unsatisfied-by-default in `CQStatus`, and require a challenger to *name* a consequence / alternative to raise them. Reuses existing `CQStatus` machinery. This is precisely the "engineering pattern adoptable independently of the philosophy" that session 11 §2 flagged — taken *with* its philosophical billing dropped (it is just a sensible default, not a "dissolution of an epistemological problem").

**5. Read factual-vs-evaluative disagreement off existing CQ metadata.**
The genuinely useful divergence-typing (FACTUAL / EVALUATIVE / STRUCTURAL, session 11 §5) needs **no netting object**: a CQ targeting a *consequence / probability* premise → FACTUAL; a *goal / value* premise → EVALUATIVE; *alternatives / feasibility* → STRUCTURAL. It is a classifier over the `targetScope` + premise-role already stored on the CQ. This keeps the part of the spec's facilitator-cockpit value that is worth having, as a read-model over existing fields.

**6. Alternatives as linked competing claims.**
When "are there better alternatives?" is answered with a named alternative, link it as a competing claim through the **existing claim + attack machinery** — not a `NettingOption`, no `argmax` fold. Lightweight comparison; the substrate already represents competing claims. *(In the cross-reference module this is the `competes` relation — deliberately **not** "rival", which is reserved for the C009 scheme-rivalry attack sense; see item 2.)*

**7. CQ-driven WHY moves (pure wiring).**
`/open-cqs` and `/legal-moves` already exist; the [`AdvDialogicalImplementationPlanning`](../../misc-documents/AdvDialogicalImplementationPlanning.txt) note already proposes pre-wiring PR's canonical CQs as WHY options with `payload.cqId`. Essentially free once item 1 lands.

**8. Cluster hygiene.**
Consolidate the `value_based_pr` / `value_based_practical_reasoning` duplicate in [`semantic-clusters.ts`](../../lib/schemes/semantic-clusters.ts) — same fix-class as the Q-018 duplicate-candidate pairs.
