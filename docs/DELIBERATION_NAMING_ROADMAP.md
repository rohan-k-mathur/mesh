## 1. Motivation

`Deliberation.title` is `String?` ‚Äî **nullable with no default**. In practice:

- **Most production creation paths leave it `NULL`.** Only the "My Arguments" personal deliberation (`title: "My Arguments"`) and seed/test scripts set a value. Discussion upgrades, comment lifts, work creation, ensure, upsert, and deepdive all create rows with `title = NULL`.
- **~8 display surfaces each invented their own fallback.** There is no single answer to "what is this deliberation called?":

  | Surface | Fallback when `title` is `NULL` |
  | --- | --- |
  | Main deliberation page (`app/deliberation/[id]/page.tsx`) | renders **host object's** name; ignores `title` entirely |
  | Loading screen (`DeliberationLoadingScreen.tsx`) | `hostName` or `"Deliberation"` |
  | Embed argument page | silent (renders nothing) |
  | Export routes | `"Untitled Deliberation"` |
  | Embed widget | `"Deliberation"` |
  | Dialogue timeline | truncated deliberation id |
  | Merge service | `"Untitled"` |
  | Dialogue-move response | `"Untitled Deliberation"` |
  | Timeline service | `undefined` (omitted) |

- **There is no write path to rename.** `app/api/deliberations/[id]/route.ts` has only `GET` (returns `{hostType, hostId}`, no title) and `DELETE`. No `PATCH`/`PUT`. A user who lands in a `NULL`-titled deliberation cannot name it.
- **Agents are blind to titles.** No MCP tool (`get_deliberation_fingerprint`, `get_synthetic_readout`, `get_contested_frontier`, `get_missing_moves`, `get_chains`, `summarize_debate`) surfaces `title`. Agents can only cite a deliberation by id.

**Goal.** Introduce an end-to-end, **non-breaking** naming feature: one canonical name resolver used by every surface, a rename write path, sensible auto-naming at creation, and agent visibility ‚Äî without backfilling or constraining existing `NULL` rows destructively.

---

## 2. Design insight: two naming concepts, one column

The nullable `title` column conflates two distinct ideas:

1. **Host-derived name** ‚Äî for host-bound deliberations (`article` ‚Üí article title, `room_thread` ‚Üí room name, `discussion`/`inbox_thread` ‚Üí topic). This is the *correct* display name for those rows and must stay authoritative.
2. **Explicit set title** ‚Äî meaningful for `free` / standalone / "My Arguments" rows that have no rich host to borrow a name from.

The non-breaking move is to **not fight existing `NULL`s**. Instead:

- Add one **canonical resolver** every surface calls.
- Add a flag distinguishing "a human deliberately named this" (sticky, never auto-overwritten) from "we auto-filled a placeholder" (upgradeable later).

---

## 3. Schema changes (additive only ‚Äî zero breaking changes)

```prisma
model Deliberation {
  title              String?    // unchanged ‚Äî stays nullable
  titleAutoGenerated Boolean    @default(false)  // NEW
  titleUpdatedAt     DateTime?                    // NEW (audit / UX)
}
```

`prisma db push` adds two columns with safe defaults ‚Äî no existing row is touched, no `NOT NULL` constraint is added, fully reversible.

`titleAutoGenerated` is load-bearing:
- Auto-generated titles can be silently **upgraded** later (e.g. when a better name becomes derivable, or by the LLM enrichment pass).
- Human-set titles are **sticky** ‚Äî never overwritten by any automated pass.

**Out of scope (per decision):** adding `@@unique([hostType, hostId])`. The index stays non-unique for now; duplicate host-bound deliberations remain technically possible and are not addressed by this roadmap.

---

## 4. The centerpiece: one resolver, used everywhere

A single helper in `lib/deliberations/` ‚Äî `resolveDeliberationName(deliberation, host?)` ‚Äî with deterministic precedence:

```
1. Human title           ‚Üí title set AND titleAutoGenerated === false
2. Host-derived name      ‚Üí article.title / room.title / discussion.topic / "My Arguments"
3. Auto-generated title   ‚Üí title set AND titleAutoGenerated === true
4. Deterministic fallback ‚Üí derived from first conclusion claim text, else "Untitled deliberation"
```

This **replaces all 8 ad-hoc fallbacks** with one import. Each current fallback string becomes a case in the resolver, so the swap is behavior-preserving for already-named rows and strictly better for `NULL` rows.

---

## 5. Auto-naming strategy

For rows that would otherwise be `NULL`, we generate a placeholder title flagged `titleAutoGenerated = true`.

- **Phase 1 (now): deterministic.** Derive from the first conclusion claim text (truncated/cleaned). Free, synchronous, no external call. Slightly clunky but always available.
- **Phase 2 (soon after): LLM enrichment.** An async pass that generates a concise summary title and overwrites **only** `titleAutoGenerated = true` rows (never human titles). Mirrors the fire-and-forget provenance-enrichment pattern already used in the quick-structured route.

> Decision (2026-05-29): ship deterministic auto-naming first, then wire up LLM enrichment soon after.

## 6. API changes

### 6.1 `PATCH /api/deliberations/[id]` (NEW)

The missing write path. Accepts `{ title }`.
- Auth-gated to creator / host-admin.
- Sets `title`, `titleAutoGenerated = false`, `titleUpdatedAt = now()`.
- Validates length / trims; rejects empty ‚Üí treat as clear-to-auto (optional).

### 6.2 `GET /api/deliberations/[id]` (EXTEND)

Return `title`, `titleAutoGenerated`, and a resolved `displayName` so clients never re-implement precedence.

- **Phase 3 follow-up:** GET `displayName` does not yet perform a polymorphic host lookup (article/post/etc. titles) ‚Äî it only resolves the free-host "My Arguments" case via `defaultHostName`. Rich host names still come from the page-level `hostName` prop. A full server-side host-name resolver can land with Phase 6/backfill if needed.
- **Phase 4:** Discussion‚Üídeliberation upgrade is now nameable. `POST /api/deliberations/ensure` accepts an optional `title` (‚â§200, trimmed); on create, precedence is: explicit human title (`titleAutoGenerated=false`) ‚Üí discussion topic prefill as a flagged auto-title (`titleAutoGenerated=true`, upgradeable later) ‚Üí `NULL`. - **Phase 4 deferrals:** (a) `spawn` route already accepts + persists `title` (human, `titleAutoGenerated=false` by default); omitted ‚Üí `NULL`, resolved to the host (article) name at read time, so no separate "manual create" modal was added (none exists in the codebase ‚Äî spawn is invoked from one-click article buttons). (b) Deterministic auto-name on omit from the *first conclusion claim* is impossible at creation time (no claims yet) ‚Äî that is the async Phase 6 enrichment; at creation we only prefill from an already-available topic.
- **Phase 5:** Surfaced `displayName` to AI agents through the MCP readouts. `DeliberationFingerprint` (`lib/deliberation/fingerprint.ts`) now selects `title/titleAutoGenerated/hostType/hostId` and computes `displayName` via `resolveDeliberationName` (with `defaultHostName`) ‚Äî **explicitly NOT part of `contentHash`** (presentation metadata, not graph content). `SyntheticReadout` (`lib/deliberation/syntheticReadout.ts`) gained a top-level `displayName` mirroring the fingerprint value. Because the readout cache is keyed by `contentHash`, a rename does not bust it; on every cache hit we re-resolve the name from the live row (`resolveDisplayName`, one cheap single-row query) and refresh both the top-level field and `fingerprint.displayName` so cached readouts never serve a stale name (also backfills old payloads written before the field existed). The compact transform (`toCompactReadout`) spreads `...readout`, so `displayName` survives. The MCP `get_deliberation_fingerprint` and `get_synthetic_readout` tools proxy these endpoints and inherit the field automatically; both tool descriptions were updated to tell agents to cite the deliberation by `displayName` rather than its id. Bumped `ORIENTATION_VERSION` 1.9.0 ‚Üí 1.10.0 and rebuilt `packages/isonomia-mcp` (clean `tsc`). **Deploy reminder:** restart Claude Desktop to pick up the rebuilt MCP server.
- **Phase 5 follow-up:** the fingerprint/readout `displayName` shares the GET-endpoint limitation ‚Äî `defaultHostName` only resolves the free-host "My Arguments" case, not polymorphic article/post host titles. A full server-side host-name resolver can land with Phase 6/backfill.
- **Phase 6:** Async LLM auto-naming. New `lib/deliberations/autoNameEnrichment.ts` exposes `enrichDeliberationName(id, { force? })` and the fire-and-forget `enrichDeliberationNameInBackground(id)` (mirrors `enrichEvidenceProvenanceInBackground`: detached via `void`/`.catch`, never throws). Gated on `isDeliberationAutoNamingEnabled` (env `DELIBERATION_LLM_AUTO_NAMING=1|true|on` + per-deliberation overrides via `DELIBERATION_LLM_AUTO_NAMING_DELIBERATIONS="<id>=on,‚Ä¶"`, same shape as the chain-essay-polish flag). Eligibility: flag on (or `force`) ‚Üí not debounced (Upstash `SET NX EX 600` lock, one LLM pass per deliberation per 10 min; skipped gracefully when Upstash absent) ‚Üí deliberation exists ‚Üí title is **not** a sticky human title (`titleAutoGenerated === false` + non-empty) ‚Üí at least `MIN_CLAIMS = 3` claims. It feeds up to 12 earliest claim texts to `gpt-4o-mini` (temp 0.3, 32 max-tokens) for a ‚â§8-word neutral title, sanitizes it (`sanitizeGeneratedTitle`: first line, strip label/quotes/asterisks/trailing punctuation, truncate to 80 chars), and falls back to `deriveAutoTitleFromClaim(firstClaim)` if the LLM is unavailable/empty. The **write is a guarded `updateMany`** (`where.OR = [{ titleAutoGenerated: true }, { title: null }]`) setting `title`, `titleAutoGenerated=true`, `titleUpdatedAt=now()` ‚Äî so a human rename that races the LLM pass always wins (returns `raced_human_title`, count 0). Triggered fire-and-forget from both argument-creation routes after the row is persisted: `app/api/arguments/quick-structured/route.ts` and `app/api/arguments/quick/route.ts` (`enrichDeliberationNameInBackground(targetDelibId)`). 12 unit tests in `__tests__/lib/deliberationAutoName.test.ts` cover the sanitizer, the flag, and every guard branch. Default-off: with the flag unset, behaviour is byte-identical to Phase 5 (the background call returns `flag_off` before any DB/LLM work).
- **Phase 6 deferrals:** (a) no MCP `rename_deliberation` write tool yet (roadmap ¬ß8 "Later"). (b) The enrichment trigger is creation-path only; a periodic backfill sweep for already-populated `NULL`/auto rows is Phase 7. (c) `sanitizeGeneratedTitle` and `isDeliberationAutoNamingEnabled` are exported for testing; `enrichDeliberationName` is also callable with `{ force: true }` for a future admin/CLI re-name endpoint.
