## 0. Goal

Make "create a deliberation from scratch" a first-class action. Today every deliberation is **host-attached**: it is minted lazily alongside some host object (article, discussion, stack, post) via [`app/api/deliberations/ensure/route.ts`](../app/api/deliberations/ensure/route.ts) or the room-chain [`app/api/deliberations/spawn/route.ts`](../app/api/deliberations/spawn/route.ts). There is **no smooth path to start a standalone deliberation** with a chosen name, visibility, and governance. This thread adds that path and, in doing so, lands the two substrate pieces (a **visibility typology** and **governance/moderator roles**) that two ratification features are blocked on.

## 1. Scope

**In (v1):**
- A canonical **visibility typology** for deliberations (the "D1 found missing" gap) + a `visibility` field on `Deliberation`.
- **Governance/moderator roles** on a deliberation (formalised on top of the existing free-text `DeliberationRole`).
- **Name-at-creation** (explicit, editable; reuses `title` + clears `titleAutoGenerated`).
- A **from-scratch create API** (`hostType: free`) distinct from the host-attached `ensure`/`spawn` paths.
- **"New deliberation" entry points** on the Agora page and the default home feed, with a create surface (name + visibility + moderators).

**Out / deferred:**
- Wiring the visibility typology into ratification's **privacy-keyed policy defaults** and the **`moderator` policy option** — those live in the ratification thread; this spec only *provides the substrate* (§8).
- Per-visibility participation rules / access enforcement beyond the deliberation-creation surface (e.g. private-room read gating across every reader) — name the typology here; enforce incrementally.
- Invitations / membership management UI beyond assigning moderators at creation.
- Migrating existing host-attached deliberations to carry an explicit visibility (backfill default = `public`; see §10).

## 2. Data model (candidate schema delta — `lib/models/schema.prisma`)

Current `Deliberation` (≈ line 4650) has `title`, `titleAutoGenerated`, `titleUpdatedAt`, `createdById`, `hostType`/`hostId`, `tags`, `roles DeliberationRole[]` — but **no `visibility`**. Visibility today is modelled inconsistently elsewhere: `AgoraRoom.visibility String @default("public")` (line ≈ 6974, `"public" | "private" | "unlisted"`), `Stack.visibility StackVisibility @default(public_closed)`, and `KbVisibility` on KB models. The D1 finding is precisely that **there is no canonical deliberation-level typology** to key policy off.

**2.1 Visibility typology (the D1 gap).** Add a canonical enum and a field on `Deliberation`.
```prisma
enum DeliberationVisibility {
  public    // listed + world-readable (Agora-surfaced)
  unlisted  // link-only; not surfaced in feeds
  private   // restricted to creator + assigned roles/moderators
}

model Deliberation {
  // …existing…
  visibility DeliberationVisibility @default(public)
}
```
- TODO: confirm the three-value typology is sufficient for the ratification "privacy-keyed defaults" need, or whether a participation axis (open / facilitated / closed) is a *separate* dimension that should NOT be conflated with read-visibility. The session flagged "participation/visibility typology" as one phrase — decide if it is one enum or two orthogonal fields.
- TODO: decide whether to retro-fit `AgoraRoom.visibility` onto this enum or leave room-visibility independent (rooms predate this; do **not** churn them in v1).

**2.2 Governance / moderators.** `DeliberationRole` already exists — `{ deliberationId, userId, role: String, invitedById }` with `@@unique([deliberationId, userId, role])` and a free-text `role` ("author"|"methodologist"|…). v1 formalises a **`moderator`** role value rather than adding a new table.
- TODO: keep `role` free-text but document the reserved governance values (`owner`, `moderator`), OR introduce a `DeliberationRoleType` enum. Lean: keep the string column (no migration churn), add a typed constant set + a helper `isModerator(deliberationId, userId)`.
- TODO: a `moderator`-capable authority check that composes with [`lib/deliberations/ownership.ts`](../lib/deliberations/ownership.ts) (`resolveHostOwnerId` returns `null` for `free`/`site` hosts, so for from-scratch deliberations `createdById` is the sole host-owner signal — moderators extend, not replace, that).

**2.3 Name-at-creation.** No schema change — reuse `title` + `titleAutoGenerated`. From-scratch create sets `title` explicit and `titleAutoGenerated: false` (mirrors the explicit-title branch already in `ensure`).

**2.4 Host binding (RESOLVED — `free`).** The **New deliberation** button creates with `hostType: free` — this is the seamless default, since `free` already has no `resolveHostOwnerId` owner (so `createdById` is the sole host-owner signal, exactly what a from-scratch deliberation wants) and is never read at eval time. Still open: `hostId` — a fresh `cuid()` (self-referential placeholder) vs `createdById` vs a dedicated `free`-host row.
- TODO: pick the `hostId` convention and document it next to the `DeliberationHostType` enum (≈ line 3475); ensure the `@@unique([hostType, hostId])` is not violated by repeated scratch creates (a per-deliberation `cuid()` `hostId` sidesteps this since each create is globally unique).

## 3. API surface

**3.1 From-scratch create** — `POST /api/deliberations/create` (new).
- Body: `{ title: string; visibility?: DeliberationVisibility; moderatorIds?: string[]; tags?: string[] }`.
- Behaviour: auth required; `hostType: free`; set `title` + `titleAutoGenerated:false`; set `visibility` (default `public`); create `DeliberationRole` rows for `createdById` (`owner`) and each `moderatorIds` (`moderator`); seed baseline Ludics designs (mirror `ensure`'s `ensureBaselineLudicsDesigns`); `emitBus("deliberations:created", { source: "create" })`; `onDeliberationCreated` trigger.
- TODO: factor the shared post-create side-effects (Ludics baseline, KG trigger, bus emit) out of `ensure`/`spawn`/`create` into one helper to avoid drift.
- TODO: decide whether to also auto-create the AgoraRoom + DebateSheet chain like `spawn` does, or keep scratch deliberations room-less until promoted.

**3.2 Relationship to existing routes.** Do **not** overload `ensure` (host-attached, idempotent on `[hostType, hostId]`) or `spawn` (article→room chain). `create` is the third, intentionally distinct entry. Cross-link all three in code comments.

## 4. UI / surfacing

**4.1 Agora entry point.** [`app/agora/page.tsx`](../app/agora/page.tsx) is a thin server wrapper around `./ui/Agora`. Add a **"New deliberation"** affordance in the Agora header/toolbar.
- TODO: locate the Agora header in `app/agora/ui/Agora.tsx` and place the button without disturbing the feed.

**4.2 Home feed entry point.** The default home is [`app/(root)/(standard)/page.tsx`](../app/(root)/(standard)/page.tsx). The repo already has sibling create surfaces — `create-room`, `create-lounge`, `create-thread` under `app/(root)/(standard)/` — so a **`create-deliberation`** page (or a modal launched from the feed composer / create menu) is the idiomatic placement.
- TODO: mirror [`app/(root)/(standard)/create-room/page.tsx`](../app/(root)/(standard)/create-room/page.tsx) for layout/auth conventions; decide page vs. modal.

**4.3 Create surface fields.** Name (editable text), visibility selector (public/unlisted/private with one-line descriptions), moderator picker (user search → `moderatorIds`), optional tags. On submit → `POST /api/deliberations/create` → redirect `/deliberation/[id]`.
- TODO: reuse the existing user-picker component used by `create-room`/groups; reuse standing visibility-selector copy if one exists.

## 5. Permissions & ownership

- Creator becomes `owner` (`DeliberationRole`), is the rename/governance authority (composes with [`ownership.ts`](../lib/deliberations/ownership.ts)).
- Moderators get governance capability (e.g. the future `moderator` ratification policy, role management) but not ownership transfer in v1.
- `private` visibility gates Agora/feed surfacing and read access to creator + assigned roles. TODO: enumerate every read path that lists deliberations (Agora events, explore, search) and apply the visibility filter — scope carefully; a partial filter is a leak.

## 6. Events

- `deliberations:created` `{ id, hostType: "free", visibility, source: "create" }` — extend the existing payload with `visibility`/`source` (already emitted by `ensure`/`spawn`).
- TODO: a `deliberations:roles:changed` event if moderator assignment is editable post-creation (likely a follow-up, not v1).

## 7. Test plan

- Unit: `create` sets `hostType:free`, explicit title, `titleAutoGenerated:false`, default `visibility:public`; moderator rows created; self as `owner`.
- Unit: visibility default + override; moderator authority helper (`isModerator`).
- Integration: a `private` deliberation does not appear in Agora/explore/search for a non-member; appears for creator + moderator.
- Regression: `ensure`/`spawn` unchanged; `@@unique([hostType, hostId])` not violated by repeated scratch creates.

## 8. What this unblocks (the ratification dependencies)

Both are **deferred in** [`ATTACK_RATIFICATION_DEV_SPEC.md`](ATTACK_RATIFICATION_DEV_SPEC.md) §1 and depend on the substrate this thread lands:
1. **Privacy-keyed policy defaults** — needs §2.1's visibility typology so `attackRatificationPolicy` can default off `visibility` (not just `hostType`). When this ships, the ratification default-seeding (its §3.1) can read `Deliberation.visibility`.
2. **A meaningful `moderator` policy option** — needs §2.2's governance roles to exist so `attackRatificationPolicy: moderator` (and the associated `REJECTED` state, its §1/§2.1) has a defined authority to check.

**Ratification v1 does not block on this thread:** its default is `hostType`-seeded (`free`→`none`, else→`single`) with an explicit override, neither of which needs visibility or moderators. Ship order is independent.

## 10. Rollout safety

- Schema delta adds `DeliberationVisibility` + a defaulted `visibility` column → existing rows backfill to `public` (no behaviour change). `db push` per repo convention; ship the field default with the column so no row is left null.
- `moderator` role formalisation is additive (string value on an existing table) — no migration risk.
- Stage: schema delta (visibility default `public`, invisible) → `create` API behind the new UI → Agora/home entry points → visibility read-enforcement (the one step with leak risk; land last, audited). Each step independently reversible.

**2026-06-14 — visibility read-enforcement (§5/§10 last stage).** Landed the leak-risk filter pass over the list/feed/search enumeration surfaces and the canonical direct-content read.

- **Helper:** `lib/deliberations/visibility.ts` — `listableDeliberationWhere(userId)` (Prisma `where`: anonymous → `public` only; signed-in → `public` ∪ own ∪ role-bound), `filterVisibleDeliberationIds(ids, userId)` (batch id gate for feed events), `canReadDeliberation(id, userId)` (direct-fetch gate: `public`/`unlisted` open, `private` members-only), `normalizeUserId(bigint|null)`. - **Membership-key note:** `cross-room-search` previously used the Supabase auth id; visibility membership is keyed on the internal `User.id`, so the route now resolves `getCurrentUserId()` for `viewerId`.
